## 🚀 Как начать

### 1. Форкните репозиторий
Нажмите кнопку **Fork** в правом верхнем углу на GitHub.

### 2. Клонируйте свой форк
```bash
git clone https://github.com/ваш-логин/undertow-client.git
cd undertow-client
```

### 3. Добавьте upstream (оригинальный репозиторий)
```bash
git remote add upstream https://github.com/daniil-verba/undertow-client.git
git fetch upstream
```

### 4. Установите зависимости
```bash
# Убедитесь, что у вас установлен Rust
rustc --version  # Должно быть 1.70+

# Установите необходимые инструменты
cargo install cargo-watch cargo-deny cargo-tarpaulin
```

### 5. Проверьте, что всё работает
```bash
cargo build
cargo test
cargo run
```

## 📖 Описание ключевых модулей

| Модуль | Назначение | Ключевые файлы |
| :--- | :--- | :--- |
| **`network`** | **Ядро P2P-сети**: управление соединениями, обход NAT, ретрансляция, DHT. | `node.rs`, `peer.rs`, `relay.rs`, `hole_puncher.rs`, `beacon_client.rs` |
| **`protocol`** | **Определения протокола**: форматы пакетов, идентификаторы пиров (`PeerId`), криптография. | `packet.rs`, `peer_id.rs`, `crypto.rs` |
| **`storage`** | **Постоянное хранилище**: профили пользователей, ключи, настройки. | `profile.rs`, `paths.rs` |
| **`dht`** | **Распределённая хеш-таблица** на основе Kademlia (поиск пиров без Beacon). | (в разработке) |
| **`crypto`** | **Криптографические примитивы**: X25519, SHA-256, AEAD (планируется). | (в разработке) |
| **`utils`** | **Вспомогательные функции**: логирование, утилиты для работы с сетью, helpers. | (в разработке) |

---

## 🔧 Взаимосвязи модулей

*   **`protocol`** — базовый слой, определяет форматы данных.
*   **`network`** — основной модуль, использует `protocol`, `crypto`, `dht` и `storage`.
*   **`storage`** — независимый модуль, использует `utils` для работы с файлами.
*   **`lib.rs`** — публичное API, скрывает внутреннюю сложность модулей.

## 📝 Правила коммитов

Мы используем [Conventional Commits](https://www.conventionalcommits.org/).

### Формат
```bash
<тип>(<область>): <описание>

[подробное описание]

[ссылка на задачу]
```

### Типы коммитов

| Тип | Когда использовать |
|-----|-------------------|
| `feat` | Новая функциональность |
| `fix` | Исправление бага |
| `docs` | Изменения в документации |
| `style` | Форматирование кода |
| `refactor` | Рефакторинг (без изменения логики) |
| `perf` | Улучшение производительности |
| `test` | Добавление тестов |
| `chore` | Вспомогательные задачи (зависимости, сборка) |
| `ci` | Настройка CI/CD |
| `revert` | Откат изменений |

### Примеры правильных коммитов

```bash
# Новая функция
git commit -m "feat(network): добавил поддержку IPv6"

# Исправление бага
git commit -m "fix(ui): исправил краш при вводе команды"

# Обновление документации
git commit -m "docs(readme): обновил инструкцию установки"

# Рефакторинг
git commit -m "refactor(app): упростил обработку событий"
```

### ❌ Примеры НЕПРАВИЛЬНЫХ коммитов

```bash
git commit -m "fix"           # ❌ Нет описания
git commit -m "ааа сломал"   # ❌ Неинформативно
git commit -m "обновил всё"   # ❌ Слишком общее
```

## 🌳 Работа с ветками

### Стратегия ветвления

```text
main  →  стабильная версия (только релизы)
  │
  └── dev  →  основная разработка
       │
       ├── feature/название-фичи   # Новая функциональность
       ├── fix/название-бага       # Исправление бага
       └── docs/название           # Документация
```

### Правила

1. **Всегда создавайте ветку от `dev`:**
   ```bash
   git checkout dev
   git pull origin dev
   git checkout -b feature/ваша-фича
   ```

2. **Называйте ветки осмысленно:**
   - `feature/add-p2p` — новая фича
   - `fix/connection-timeout` — исправление бага
   - `docs/update-readme` — документация

3. **Одна ветка = одна задача**

4. **После слияния ветка удаляется**

### Пример рабочего процесса

```bash
# 1. Создаём ветку для фичи
git checkout dev
git checkout -b feature/add-p2p

# 2. Работаем, делаем коммиты
git add .
git commit -m "feat(network): добавил P2P соединение"
git commit -m "test(network): добавил тесты для P2P"

# 3. Пушим в свой форк
git push origin feature/add-p2p

# 4. Создаём Pull Request на GitHub
```

## 🔄 Создание Pull Request

### Чек-лист перед созданием PR

- [ ] Ветка создана от **актуального `dev`**
- [ ] Все коммиты соответствуют [Conventional Commits](#правила-коммитов)
- [ ] Код проходит `cargo fmt` и `cargo clippy`
- [ ] Все тесты проходят: `cargo test`
- [ ] Добавлены тесты для новой функциональности
- [ ] Обновлена документация (если нужно)
- [ ] Ветка запушина в ваш форк

### Создание PR

1. Зайдите на GitHub в свой форк
2. Нажмите **"Compare & pull request"**
3. Выберите:
   - **base:** `daniil-verba/undertow-client:dev`
   - **compare:** `ваш-логин/undertow-client:feature/ваша-фича`
4. Заполните шаблон PR:

```markdown
## Описание изменений
Кратко опишите, что вы сделали.

## Связанные задачи
Closes #42 (если есть)

## Тип изменений
- [ ] Новая функциональность (feat)
- [ ] Исправление бага (fix)
- [ ] Документация (docs)
- [ ] Рефакторинг (refactor)
- [ ] Тесты (test)

## Как тестировать?
1. Шаг 1
2. Шаг 2

## Чек-лист
- [ ] Код проходит `cargo fmt`
- [ ] Код проходит `cargo clippy`
- [ ] Тесты проходят `cargo test`
- [ ] Документация обновлена
```

### Пример хорошего PR

```markdown
## Описание изменений
Добавлена поддержка IPv6 для P2P соединений.

## Связанные задачи
Closes #42

## Тип изменений
- [x] Новая функциональность (feat)

## Как тестировать?
1. cargo run -- --ipv6
2. Подключиться к пиру по IPv6 адресу

## Чек-лист
- [x] Код проходит `cargo fmt`
- [x] Код проходит `cargo clippy`
- [x] Тесты проходят `cargo test`
- [x] Документация обновлена
```

## 👀 Code Review

После создания PR кто-то из мейнтейнеров проведёт ревью.

### Что мы проверяем

1. **Соответствие кода стилю проекта**
2. **Наличие тестов**
3. **Отсутствие регрессий**
4. **Качество документации**

### Что делать, если просят исправить

```bash
# Вносите изменения в ту же ветку
git add .
git commit -m "fix: исправления по ревью"
git push origin feature/ваша-фича
```

После пуша PR автоматически обновится.

## 🧪 Тестирование

### Запуск всех тестов

```bash
cargo test
```

### Запуск конкретного теста

```bash
cargo test test_name
```

### Запуск с выводом логов

```bash
cargo test -- --nocapture
```

### Проверка покрытия кода

```bash
cargo tarpaulin --ignore-tests
```

### Требования к тестам

- **Новая функциональность** → должны быть тесты
- **Исправление бага** → должны быть тесты, воспроизводящие баг
- **Рефакторинг** → тесты должны проходить без изменений

## 📚 Документация

### Обновление README

Если ваше изменение затрагивает:
- Новые команды
- Изменение конфигурации
- Новые требования

Обновите `README.md` и `README.ru.md` (если есть).

### Обновление CHANGELOG

1. При каждом релизе создавай новый раздел с версией и датой.
2. Сортируй изменения по важности: `Added` > `Changed` > `Fixed` > `Deprecated` > `Removed` > `Security`.
3. В `Unreleased` собирай все изменения, которые уже есть в `dev`-ветке, но еще не вошли в релиз.
4. После релиза переноси всё из `Unreleased` в новый раздел с версией.
5. Каждое изменение описывай одной строкой в формате: `- Описание изменения (#номер-issue)`

## Примеры хороших записей

```markdown
### Added
- Добавлена поддержка IPv6 для P2P-соединений (#42)
- Добавлен CLI-аргумент `--config` для указания пути к конфигу (#15)

### Fixed
- Исправлена паника при отключении Beacon во время передачи (#37)
- Исправлен баг с дублированием сообщений в Relay (#28)

### Changed
- Обновлена зависимость `tokio` с 1.30 до 1.35 (#41)
```

### Документация кода

Используйте комментарии `///` для публичных функций и структур.

## ❓ Часто задаваемые вопросы

### "Я новичок в Rust. Могу ли я участвовать?"

✅ Лучше создать Issue сначала, чтобы обсудить, нужна ли эта фича.

### "Что делать, если я застрял?"
Создайте PR с пометкой `[WIP]` (Work In Progress) и задайте вопрос.

### "Как часто обновлять форк?"
```bash
git fetch upstream
git checkout dev
git merge upstream/dev
git push origin dev
```
